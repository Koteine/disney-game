# План облегчения кода (без поломки игровых механик)

Этот план заточен под текущую структуру проекта: основной код в `index.html` + небольшие модули в `js/modules/*`.

## 1) Главный принцип

Сейчас самая большая точка риска — огромный `index.html` с несколькими доменами сразу (игра, розыгрыш, админка, работы, ивенты, инвентарь).

**Цель:** уменьшать связанность и размер файла поэтапно, оставляя публичные API-имена стабильными (через `window.*` на переходный период).

---

## 2) Что даст быстрый выигрыш (1–2 итерации)

### 2.1. Вынести крупные домены из `index.html` в отдельные файлы

Приоритет выноса:

1. `works` (загрузка/рендер работ)
2. `tickets + wheel`
3. `admin page`
4. `events`

Минимальный безопасный подход:

- Скопировать блок домена в `js/modules/<domain>.js`.
- В конце модуля экспортировать публичные методы в `window` (только те, что реально вызываются из HTML через `onclick`).
- В `index.html` оставить только загрузку скрипта.
- После каждого выноса прогонять smoke-check по ключевым вкладкам.

### 2.2. Убрать inline `onclick` постепенно

Вместо:

- `onclick="..."`

Сделать:

- `data-action="..."` + единый делегированный обработчик на контейнер.

Польза:

- легче искать связи,
- меньше риск «тихой» потери функции при рефакторинге,
- проще покрывать тестами.

### 2.3. Централизовать константы

Вынести в `js/config/constants.js`:

- Firebase paths (`board`, `tickets`, `player_season_status` и т.д.),
- тайминги (`ONLINE_HEARTBEAT_MS`, `DUEL_PENDING_MS` ...),
- роли/идентификаторы.

Это уменьшит дубли и ошибки в строковых путях к БД.

---

## 3) Что снизит баги сильнее всего (средний горизонт)

### 3.1. Ввести слой доступа к Firebase

Создать thin-adapter, например `js/services/db.js`, где будут функции:

- `getBoard()`, `subscribeBoard(cb)`,
- `updatePlayerInventory(userId, patch)`,
- `appendNewsItem(payload)` и т.д.

Почему это важно:

- доменная логика перестаёт напрямую знать структуру путей,
- меньше шансов сломать приложение опечаткой в `db.ref(...)`.

### 3.2. Разделить состояние на доменные store-объекты

Вместо россыпи глобальных переменных:

- `window.gameState`,
- `window.worksState`,
- `window.eventsState`.

Даже без фреймворка это уже сильно упрощает поддержку.

### 3.3. Единый слой уведомлений и ошибок

Сейчас в коде много `alert(...)` и разрозненного `console.error(...)`.

Сделать:

- `notify.info/warn/error`,
- `reportError(scope, err, meta)`.

Это упрощает отладку и дает одинаковое UX-поведение.

---

## 4) Безопасная стратегия миграции

Для каждого шага применять правило:

1. **Сначала** копируем логику в новый модуль, не удаляя старую.
2. Переключаем вызовы на модуль.
3. Проверяем smoke-сценарии.
4. **Только потом** удаляем старую реализацию.

Это защищает от ситуации, когда при удалении «дубликатов» случайно удаляется рабочий единственный путь исполнения.

---

## 5) Минимальный smoke-набор после каждого шага

Проверять вручную/автотестом:

1. Вход в приложение и отображение профиля.
2. Бросок кубика и открытие клетки.
3. Рендер инвентаря.
4. Загрузка работы (игрок).
5. Просмотр/рендер работ (админ).
6. Переходы по вкладкам, особенно `rules`, `works`, `admin`, `wheel`.
7. Запуск админ-события.

---

## 6) Что делать НЕ нужно

- Не делать «большой взрыв» с переносом всего `index.html` за один PR.
- Не удалять утилиты/функции, пока не найдено **все** места вызова (включая inline-обработчики и `window`-ссылки).
- Не смешивать функциональные изменения и архитектурную чистку в одном большом коммите.

---

## 7) Практичный roadmap по PR

- **PR-1:** `works`-домен (вынос + без изменения поведения).
- **PR-2:** `admin page`-домен.
- **PR-3:** `tickets/wheel`-домен.
- **PR-4:** `events`-домен.
- **PR-5:** `db service + constants` + замена прямых `db.ref` в первых двух доменах.
- **PR-6:** удаление остаточного dead-code после стабилизации.

Так вы реально облегчите код и снизите баги, но без резкой потери функций.
